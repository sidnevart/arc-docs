# Desktop для новичков

Новый desktop теперь строится вокруг разговора с агентом и сессий, а не вокруг ручного task manager и не вокруг файлов.

## Новый главный путь

Теперь базовый сценарий такой:

1. открыть проект через welcome chooser
2. при необходимости нажать `Настроить ARC`
3. перейти в `Чат`
4. выбрать встроенного агента в секции `Агент проекта` слева
5. прочитать короткое описание агента перед началом диалога
6. написать цель простыми словами
7. при необходимости нажать `Добавить файлы`, чтобы положить свои файлы в проект без терминала
8. смотреть схемы, демо и симуляции прямо в разговоре
9. открывать `Детали чата` справа только по запросу
10. если хочется увеличить или уменьшить чат, открыть `View -> Display...` и выставить процент масштаба

## Что означают экраны

## `Чат`

Это главный экран.

Здесь ты:

- общаешься с агентом
- выбираешь тему слева
- видишь короткое creator-authored описание выбранного агента в начале нового диалога
- видишь thread по центру
- видишь схемы, демо и симуляции прямо внутри разговора
- открываешь документы, файлы и детали справа в drawer только когда они реально нужны
- при желании меняешь масштаб чата в процентах через отдельное окно `Display`, не захламляя сам chat UI отдельным переключателем

То есть разговор теперь сам становится рабочим входом в систему.

Короткое описание агента должно отвечать на три вещи:

- что этот агент умеет лучше всего
- как лучше формулировать запрос
- чего он по умолчанию не делает

Это описание должно задаваться создателем агента, а не собираться на лету из случайных UI строк.

Отдельного экрана `Сессии` больше не нужно.

Прошлые сессии теперь выглядят как обычные темы в левой колонке:

- прошлые разговоры
- связанные планы и запуски
- материалы, демо и документы
- поиск и продолжение нужной сессии

Именно оттуда теперь проще всего понять:

- что уже делалось
- что получилось
- что стоит продолжить
- какой прошлый контекст подтянуть в новый разговор

## Материалы сессии

Отдельного экрана `Результаты` больше не нужно.

Теперь результаты живут прямо внутри каждой сессии как карточки:

- ответ
- план
- схема
- симуляция
- демо
- документ
- проверка
- изменения

Если агент создал мини-сайт, демо или симуляцию, ARC старается показать это прямо в разговоре как inline preview или live miniapp block.

Важно: source files такого результата сохраняются вместе с сессией. Даже если live preview был остановлен по idle policy или вручную, ARC должен уметь заново поднять demo/simulation из сохранённого artifact прямо из чата.

Документы, длинные ответы, связанные файлы и технические детали при этом открываются справа в drawer, чтобы не ломать сам ритм чата.

По умолчанию приоритет такой:

1. показать визуальный результат прямо в thread
2. дать кнопку `Открыть в ARC`
3. открыть документы и детали справа
4. дать пользователю кнопку `Остановить`, если это живое локальное приложение

Если миниапп уже был остановлен, вместо мёртвого preview ARC должен показать состояние `Stopped, can restart` и снова поднять preview перед `Открыть в ARC`.

Если именно visual-turn сорвался из-за transient Codex runtime сбоя уровня `failed to refresh available models`, ARC теперь не падает сразу после первого таймаута. Для reply-only chat turn он автоматически делает ещё одну попытку с увеличенным timeout budget и пишет рядом `turn-XXX.retry.{json,md}` артефакты, чтобы было видно, что это был recoverable runtime stall, а не обычная логическая ошибка ответа.

`Детали чата` теперь работают как один понятный drawer:

1. `Сейчас запущено` показывает активные демо и симуляции текущей темы
2. `Материалы чата` показывают ответы, документы, схемы, демо и симуляции текущей темы
3. `Файлы` показывают обычные файлы проекта и отдельно ARC-managed uploads
4. запуск или остановка демо синхронизируются между карточкой в сообщении, fullscreen ARC window и этим drawer

## Загрузка материалов без терминала

Если у тебя есть PDF, Markdown, изображения, заметки или любые другие рабочие материалы, их теперь можно загрузить через кнопку `Загрузить материалы` прямо в `Чате`.

Что происходит дальше:

1. ARC сохраняет файлы в `.arc/materials/uploads/...`
2. эти файлы сразу становятся частью ARC context
3. если разговор уже открыт, ARC может связать загрузку и с текущей сессией
4. агент начинает видеть эти материалы без ручного копирования путей

ARC-managed uploads можно удалить из секции `Файлы` внутри `Детали чата`, не трогая обычные файлы проекта.

## Display scale

Размер chat workspace теперь регулируется отдельно от основного UI shell.

- диапазон: `60%–130%`
- шаг: `5%`
- native: `View -> Display...`
- preview: mirrored Display surface

Scale влияет на rail, thread, composer и drawer content, но не обязан одинаково масштабировать весь остальной desktop shell.

Это особенно полезно для непродвинутых пользователей, которым неудобно разбираться в файловой структуре проекта.

## Обучение через `Study`

Это не отдельный сложный экран, а стиль помощи агента и набор стартовых запросов внутри того же чата.

Первый demo-сценарий — объяснение клеточного дыхания через текст, схему и мини-симуляцию.

Это важная часть `Study`: он не только отвечает текстом, но и может собрать:

- диаграмму
- мини-симуляцию
- туториал
- короткий self-check

## `Настройки`

Здесь живут:

- путь к проекту
- доступные агенты
- логи
- быстрый переход к смене проекта

Настройка масштаба и смена языка намеренно не вынесены в сам chat viewport.
Для этого есть системное меню приложения:

- `View -> Display...`
- `ARC -> Language`
- `ARC -> Settings`
- внутри `Display` можно выбрать процент масштаба, например 90%, 100% или 110%

## Что изменилось по сравнению со старой моделью

Теперь основной принцип такой:

- выбери проект
- выбери встроенного агента в одном project-level picker сверху
- пиши агенту сразу
- смотри на статус агента прямо в разговоре
- читай ответы агента как normal markdown, а не как сырые plain-text куски
- система сама создаёт и продолжает сессию
- диаграммы и демо смотри прямо в разговоре
- документы и детали открывай справа внутри этой же сессии
- файлы открывай только когда они действительно нужны

## Когда открывать файлы

Не в начале работы.

Открывай файл только если:

- в результате появились изменения
- нужно посмотреть diff
- хочется понять, почему агент поменял именно это место

То есть файл теперь не “отдельный продукт”, а объяснение результата.

## Встроенный local runtime

Desktop умеет сам поднимать простые локальные preview apps без обязательного Docker у пользователя.

Это выглядит так:

1. агент создаёт demo, мини-сайт или симуляцию
2. ARC запускает локальный preview как управляемый процесс
3. preview показывается прямо внутри приложения
4. при желании можно нажать `Открыть в ARC`, а уже из demo window открыть браузер
5. пользователь всегда видит, что сейчас запущено, и может это остановить вручную

Временные preview apps автоостанавливаются по policy. Если агент не уверен, оставлять ли что-то запущенным, решение остаётся за пользователем через UI.

## Что делать, если проект старый

Если проект был создан на более старом scaffold ARC, можно обновить локальную структуру так:

```bash
go run ./cmd/arc workspace repair --path .
```

Эта команда добавит недостающие части `.arc`, например skills, hooks и provider guidance, не затирая пользовательские файлы проекта.
